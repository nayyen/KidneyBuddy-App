<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| client -> /api/anomaly | Feedback/acknowledge requests; must be scoped to req.user.id (IDOR) |
| user free-text -> Groq prompt | fluid/activity/lab catatan flows into the explanation prompt as untrusted data |
| backend -> user devices | Emergency push fan-out carries alert text |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-05 | Elevation of Privilege | `/api/anomaly/:id` feedback/acknowledge | mitigate | `authenticate` on every route + repository functions scope by `req.user!.id`; no bare id mutation. |
| T-05-06 | Tampering | Prompt injection via user free-text in explanation prompt | mitigate | System prompt instructs the model to treat user fields as data not instructions; D-20 forbidden-phrase validator is a second layer for tinggi-severity output. |
| T-05-07 | Tampering | Emergency-modal client bypass | mitigate | Acknowledge persists status server-side (aktif->dibaca); modal re-check uses the server `active-high-severity` query, not client-persisted flags (D-07). |
| T-05-08 | Information Disclosure | GROQ_API_KEY / raw bodies in logs | mitigate | pino calls never log the key or raw Groq request/response bodies (notification.service.ts convention). |
</threat_model>
