## Color

Base palette inherited from Phases 2-4. Phase 5 adds no new hex tokens — it locks down which of the two existing DESIGN_SYSTEM alert specs applies to which severity, which was previously ambiguous.

### 60/30/10 Model

| Role | Value | Usage |
|------|-------|-------|
| Dominant (60%) | `#fdf9f3` (cream) | Page background — `--color-background` (inherited) |
| Secondary (30%) | `#ffffff` (white) | Cards: AI content cards, alert cards, alert history list items, modal content surface |
| Accent (10%) | `#2a9d8f` (teal) | See reserved-for list below |
| Warning / Status | `#ef9f27` (amber) | Narrow, specific reservation — see below. NOT used for anomaly severity. |
| Destructive | `#d4183d` | High-severity anomalies ONLY (emergency modal) |

**Accent (`#2a9d8f`) reserved for — Phase 5 additions in bold:**

1. Primary button background (inherited)
2. Form input focus ring (inherited)
3. Lab trend chart line/dot (inherited Phase 3)
4. **AI content card icon color** (Sparkles/Lightbulb/FlaskConical/Leaf icons inside the 30×30px icon container)
5. **AI badge chip text color** ("AI" label)
6. **Normal-severity anomaly alert card icon + accent** (per Alert Variant Lock below)
7. **"Buat Ringkasan" / "Buat Ulang Ringkasan" button** (teal, matches primary button pattern)
8. **Active status badge on `/notifikasi`** ("Aktif")

**Amber (`#ef9f27`) reserved for — Phase 5, narrow scope only:**

1. **Unread/unacknowledged alert indicator dot** on Beranda alert cards and on the notification bell icon badge — this is the ONLY new amber usage in Phase 5, chosen because it reuses the already-established "status dot" convention (Card Kegiatan `Berlangsung` = amber dot) rather than inventing new meaning for amber.

Amber is explicitly NOT used for anomaly severity, AI content, or any button in this phase — do not use amber as a "medium severity" color; severity is strictly binary (normal = teal, tinggi = red) per D-02.

**Destructive (`#d4183d`) reserved for — Phase 5, exclusively:**

1. Emergency modal (background `#fdecee`, icon `#d4183d`, per Alert Darurat spec)
2. High-severity status text/badge wherever a `tinggi` alert is referenced (e.g., in `/notifikasi` history, in the report's Anomali section)
3. NOT used for form validation errors in this phase (none exist) and NOT used for the "Tidak Relevan" feedback button (that is a neutral secondary action, not destructive — see Copywriting Contract)

### Alert Variant Lock (resolves ambiguity — read before implementing any anomaly UI)

DESIGN_SYSTEM_KidneyBuddy_v3.md defines exactly two alert component specs: **"Alert Darurat"** (red) and **"Alert AI / Info"** (teal gradient). Per canonical_refs, Phase 5 must use these exact specs and must NOT invent a third color for anomaly severity. This resolves as:

| Severity (D-02, fixed by type) | Visual spec | Where it renders |
|---|---|---|
| `tinggi` (CAPD abnormal effluent, >2 missed schedules/day) | **Alert Darurat** — bg `#fdecee`, border-left 3px `#d4183d`, icon container bg `#fbd9dd` icon `#d4183d` | `EmergencyAnomalyModal` (full-screen, non-dismissable) |
| `normal` (≥30% fluid-output decline, deviated fluid intake) | **Alert AI / Info** — bg gradient `linear-gradient(135deg, #f0faf9, #e0f5f2)`, border `0.5px solid #cfe8e4`, icon container bg `rgba(255,255,255,.7)` icon `#2a9d8f` | Alert card on Beranda + `/notifikasi` history list item |

**Architectural decision (flagged for planner confirmation):** `CAPDEffluentBanner.tsx` (Phase 2, FLUID-03) is **superseded, not duplicated, by `EmergencyAnomalyModal`** for the CAPD-abnormal-condition trigger. Per CONTEXT.md code_context, the modal is explicitly "a more forceful variant of this same interaction model" and reuses the same backend acknowledge pattern. Rendering both the old banner AND the new modal for the same event would be confusing and redundant. **Remove the `<CAPDEffluentBanner>` render from `beranda/page.tsx`** and let the new unified `EmergencyAnomalyModal` handle all `tinggi`-severity events (CAPD abnormal effluent AND missed-schedule anomalies) through one consistent mechanism. `CAPDEffluentBanner.tsx` itself is not deleted (avoid breaking other imports/tests without confirmation) but is no longer rendered in the Beranda page tree.

**All four AI content cards use the "Alert AI / Info" gradient background** (`linear-gradient(135deg, #f0faf9, #e0f5f2)`, border `0.5px solid #cfe8e4`) — this is the established "this is AI-generated" visual signal already used by `AiPlaceholderCard`. Daily summary, weekly insight, lab analysis, and lifestyle suggestion all share this exact background treatment; do not give any of them a unique color.

**No visual differentiation between an LLM-generated explanation and a static fallback template (D-20).** This is intentional — the safety net must be invisible to the end user (only logged server-side). Do not add a "fallback mode" badge or different styling when D-20's forbidden-phrase check swaps in a static template.

### Destructive Actions

Phase 5 has no data-deleting destructive actions. The closest analog — marking an alert "Tidak Relevan" — is feedback, not deletion, and uses a **neutral secondary button style**, not the destructive red button style. Do not style "Tidak Relevan" as if it were a delete/destroy action.

| Element | Label | Style |
|---------|-------|-------|
| Feedback: positive | "Relevan" | Outline pill, teal border `#2a9d8f`, `ThumbsUp` icon, becomes filled teal when selected |
| Feedback: negative | "Tidak Relevan" | Outline pill, neutral border `#cfe8e4` text `#7a8c8a`, `ThumbsDown` icon, becomes filled `#7a8c8a`/muted (NOT red) when selected |
