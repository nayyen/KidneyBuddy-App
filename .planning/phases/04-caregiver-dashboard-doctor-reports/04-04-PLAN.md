## Phase Goal

**As a** patient about to see my doctor, **I want to** pick a date range, add an optional note, and print or save a clean PDF report of my tracking data, **so that** I can hand my doctor an accurate summary without exposing or editing the underlying records.

> Source: ROADMAP Phase 4 Success Criteria 3 & 4. Grounded in CONTEXT.md D-04/D-05/D-06/D-07/D-09 and 04-UI-SPEC Screens 2 & 3.

<objective>
REPORT-02 (+ REPORT-01 rendering) vertical slice: the two-screen report flow. `/laporan` lets the user choose a date range and write an optional doctor note; `/laporan/preview` fetches `GET /api/report` and renders the four report sections in a print-friendly layout, with `window.print()` producing the PDF. The doctor note lives only in component state and a URL param — never the database (D-06).

Purpose: This is the user-facing half of the doctor report. It consumes the 04-03 endpoint and the 04-01 textarea. Print correctness (hiding the AppShell, A4 layout) and URL-encoding the note (Pitfall 3) are the failure-prone areas.
Output: laporan page + date-range selector, preview page + content + four sections, print CSS.
</objective>
