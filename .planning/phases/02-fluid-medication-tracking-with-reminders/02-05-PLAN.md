<objective>
Build the reminder backend that all three reminder types share: extend reminder_schedule with the Phase-2 columns (dose, jenis_obat, foto_obat, konsentrasi_capd, follow_up_sent, last_notification_sent_at), add the medication_log table and its confirm/missed state machine, build the reminders + medicationLog services/controllers/routes (with multer photo upload), and wire the therapy-change hook so switching method preserves medication reminders while auto-deactivating CAPD/HD ones.

Purpose: REMIND-01/05/06 (create reminders per type), REMIND-03 (confirm a dose), and REMIND-07 (therapy-change behavior) are the data + business layer the cron dispatch (02-06) and reminder UI (02-07) build on. The reminder_schedule extension must be a non-destructive ALTER.

## Phase Goal (user story)

**As a** patient on any therapy type, **I want to** create medication, CAPD exchange, or HD schedule reminders and confirm doses I have taken, **so that** my reminder schedule reflects my real regimen and my adherence is recorded.

## Artifacts this plan produces

- Extended table: `reminder_schedule` +dosis, +jenis_obat, +foto_obat, +konsentrasi_capd, +follow_up_sent, +last_notification_sent_at
- New table: `medication_log` (id, user_id FK, reminder_id FK, nama_obat, dosis, jenis_obat, status, waktu_pengingat, waktu_konfirmasi, index on user_id+waktu_pengingat)
- New repositories: reminderSchedule (extended: list/update/delete/deactivateTherapySpecific), medicationLog.repository.ts
- New services: reminders.service.ts (createReminder/list/update/remove for obat|capd|hd), medicationLog.service.ts (confirm, findTodayUnconfirmed)
- New endpoints: `POST/GET/PATCH/DELETE /api/reminders`, `POST /api/medication-log/confirm`, `GET /api/medication-log/today`, `GET /api/reminders/next`
- New: backend/src/lib/upload.ts (multer diskStorage, 10MB, image/jpeg|png)
- Modified: profile.service.ts therapy-change hook (REMIND-07); docker-compose.yml backend uploads volume
- Backend dependency: `multer`, `@types/multer`
