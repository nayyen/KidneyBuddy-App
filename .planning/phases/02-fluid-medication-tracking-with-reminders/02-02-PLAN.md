<objective>
Build the load-bearing backend infrastructure the entire reminder system depends on: per-device VAPID push subscriptions (one-to-many from user, unique on endpoint), the `sendToAllDevices` fan-out, the VAPID web-push client, and the AES-256-GCM encryption helpers used by fluid/medication free-text columns. Also harden the API with helmet and mount the push routes.

Purpose: NOTIF-02 (per-device subscriptions) and REMIND-08 (caregiver multi-device) cannot be retrofitted after the schema ships — the per-device model and fan-out must be correct from the first row. Encryption (NFR-02) must exist before any sensitive health column is written in plan 02-04.
Output: encryption.ts, webPushClient.ts, pushSubscriptions schema + repository, push + notification services, push routes mounted, helmet enabled, and unit tests for encryption / per-device upsert / fan-out.
</objective>

## Phase Goal (user story)

**As a** patient whose caregiver is logged in on a second phone, **I want to** have each device register its own push subscription independently, **so that** a reminder reaches both devices and neither silently overwrites the other.

| Boundary | Description |

| client→/api/push | Untrusted PushSubscription JSON crosses here; must be auth-scoped to the caller |
| backend→external push service | VAPID-signed messages leave the trust boundary |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-02-01 | Tampering | POST /api/push/subscribe registering another user's device | mitigate | authenticate middleware required; subscription stored under req.user.id only |
| T-02-02-02 | Information Disclosure | VAPID_PRIVATE_KEY / ENCRYPTION_KEY leak via logs | mitigate | keys loaded once from env, validated at startup, never passed to pino/console or DB |
| T-02-02-03 | Information Disclosure | fluid/medication free-text readable in DB | mitigate | app-layer AES-256-GCM (not pgcrypto); key never transits SQL; rotation documented in .env.example |
| T-02-02-04 | Spoofing | forged subscription endpoint to redirect a user's pushes | mitigate | unique on endpoint + auth scoping; 410 cleanup removes dead endpoints |
| T-02-02-SC | Tampering | npm installs (web-push, helmet) | mitigate | blocking-human legitimacy checkpoint (Task 1) verifying npmjs publishers |
</threat_model>
