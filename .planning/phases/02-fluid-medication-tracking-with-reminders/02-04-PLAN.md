<objective>
Deliver the fluid-tracking vertical slice end-to-end: the fluid_log table (with an encrypted free-text column), the create + daily-balance backend, and the frontend to log an entry and immediately see the recalculated delta on Beranda — including the non-dismissable CAPD effluent alert, retroactive late entries, and an offline IndexedDB queue. Output: backend fluid schema/repo/service/controller/route + balance, [BLOCKING] schema push, fluid.service tests, and the full Beranda + Catatan/Cairan UI with the CatatCairan sheet/form and offline queue.
</objective>

## Phase Goal (user story)

**As a** CKD patient, **I want to** log every fluid I take in or pass out and instantly see my running daily balance, **so that** I can manage my fluid limits and catch dangerous CAPD effluent conditions early.

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-02-04-01 | Information Disclosure | reading another user's fluid log | mitigate | all repo queries filter WHERE user_id = req.user.id; routes behind authenticate |
| T-02-04-02 | Information Disclosure | catatan health note readable in DB | mitigate | encrypt() before INSERT (02-02 AES-256-GCM); decrypt only in service after SELECT |
| T-02-04-03 | Tampering | negative/garbage volume or invalid enum | mitigate | zod validation in fluid.service (volume positive, enum tipe/sumber/kondisiKeluar) |
| T-02-04-04 | Tampering | client recomputing balance (dual truth) | mitigate | balance computed only in backend; client renders the returned delta |
| T-02-04-05 | Repudiation | CAPD abnormal-condition acknowledgment lost | mitigate | banner action POSTs an acknowledgment so the patient's "saya mengerti" is recorded |
</threat_model>
