<objective>
Build the lab-results backend core for manual entry, listing, and soft-delete archive (LAB-02, LAB-04): the `lab_results` table (typed JSONB `parameter_lab`), repository, service (Zod + AES-256-GCM encryption of `catatanTambahan`), controller, routes, mount, and DB push. File upload and trend queries are added in 03-05; the chart UI in 03-06.

Purpose: Manual lab entry is the primary data source for the trend chart (D-11) and the simplest lab capability — a patient can record a parameter value and never lose it (archive-only, LAB-04).
Output: `/api/lab-results` POST / GET / PATCH :id/archive endpoints with `lab_results` pushed to Postgres and green unit tests.
</objective>

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-03-11 | Tampering | archive another user's lab result (IDOR) | mitigate | `archiveById` and all queries filter `WHERE id = ${id} AND user_id = ${userId}`; controllers use req.user!.id |
| T-03-12 | Information Disclosure | catatanTambahan health note at rest | mitigate | AES-256-GCM encrypt before INSERT; decrypt only on read |
| T-03-13 | Tampering | permanent deletion of lab history (LAB-04) | mitigate | No DELETE route or repository delete exists; archive is status="diarsipkan" only; verified by grep gate (0 `.delete(`) |
| T-03-14 | Information Disclosure | parameter_lab medical values unencrypted | accept | JSONB must stay queryable for trend chart (D-11); access is gated by JWT auth + per-user WHERE; documented tradeoff (RESEARCH A2) |
| T-03-06 | Input Validation | lab payload | mitigate | Zod `createManualLabSchema` validates types, positivity, required fields |
| T-03-03 | Spoofing | route auth | mitigate | `authenticate` on every /api/lab-results route |
</threat_model>
