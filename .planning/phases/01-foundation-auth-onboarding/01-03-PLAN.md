<objective>
Add the forgotten-password recovery slice (AUTH-06): a user requests a reset link by email, receives a time-limited single-use token, and sets a new password through a tokenized link, then logs in with it.

Purpose: Without this, a user who forgets their password is permanently locked out (only soft-delete exists, no recovery path) — the exact gap surfaced during discuss-phase that motivated adding AUTH-06 (D-01). Reset tokens must be single-use and time-limited (CONTEXT.md Claude's Discretion: ~1 hour validity).
Output: forgot-password + reset-password flow end-to-end; new password works for login; old reset token rejected after use.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Browser -> Backend (reset request) | Untrusted email/token input; enumeration + brute-force surface. |
| Email channel | Reset link transits an out-of-band channel; token must be unguessable + short-lived. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-10 | Information Disclosure | account enumeration via forgot-password | mitigate | Identical generic response for existing/non-existent emails; no token created for unknown emails. |
| T-01-11 | Spoofing/Elevation | reset token guessing/replay | mitigate | crypto.randomBytes token; store only its hash; single-use (usedAt) + ~1h expiry; revoke refresh tokens on reset. |
| T-01-12 | Tampering | reset password strength | mitigate | newPassword validated server-side with the same zod rule as registration (never trust client). |
| T-01-13 | Repudiation | reset events | accept | Dev email service logs the reset URL; structured audit logging of reset events deferred to later hardening (academic MVP). |
| T-01-SC | Tampering | npm installs (email provider, if any added) | mitigate | MVP uses Node built-in crypto + pino logging (no new package). If a mail package is added later it must pass the Package Legitimacy Gate; no [ASSUMED]/[SUS]/[SLOP] packages introduced here. |
</threat_model>
