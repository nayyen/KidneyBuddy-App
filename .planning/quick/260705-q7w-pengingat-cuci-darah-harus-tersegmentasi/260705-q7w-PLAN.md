<objective>
Fix a data-scoping bug: cuci-darah (dialysis) reminders are not correctly segmented by the user's active therapy method (`metodeTerapiAktif`: CAPD / HD / Transplantasi), while medication (obat) reminders must remain unaffected.

Root cause: therapy scoping is currently implemented DESTRUCTIVELY. On therapy change, `profile.service.changeTherapyMethod` calls `reminderScheduleRepository.deactivateTherapySpecific`, which sets `aktif = false` on the old therapy's reminders. But `aktif` is ALSO the user-facing enable/disable toggle (see `ReminderItem.tsx` line 163). This conflation means (a) switching back can never restore reminders — the system can't tell a therapy-deactivation apart from a manual disable — and (b) the /pengingat list never filters by therapy at all, so old-therapy reminders still appear.

The `jenis` field (`'obat' | 'capd' | 'hd'`) already exists on `reminder_schedule` — NO schema migration is needed. The fix replaces the destructive `aktif` mutation with a pure, non-destructive query-time filter driven by `metodeTerapiAktif` vs `jenis`, applied consistently across the reminder list, the beranda card, next-upcoming, and push dispatch. The user's `aktif` toggle is preserved as an orthogonal concern.

Purpose: Guarantee therapy-correct reminder visibility with lossless, reversible switching — core to KidneyBuddy's "never miss a dialysis session" value.
Output: Therapy-scoped backend queries + Transplantasi-gated beranda card, with tests.
</objective>
