## Decisions Made

See frontmatter `key-decisions`. Most notably: the plan's `<success_criteria>` assumed "file JSON selain fluid Budi/Sari byte-identical setelah regenerate" — this is architecturally impossible given the generator's single shared `SeededRng` instance consumed sequentially across all sections in file order. Any change to rng-call count in an earlier section (sari/budi fluid) necessarily shifts the random stream for every later section (daily_activities, lab-results uuids, AI caches, community posts/replies, education). Investigated and confirmed this is benign:
- Sections generated **before** the fluid blocks (`users`, `reminder_schedule`, medication/dialysis logs, Lukman's fluid) are byte-identical (confirmed via `git diff --stat`, empty).
- `anomaly-alerts.json` is fully hardcoded literal data (no rng calls) — byte-identical, confirmed unchanged.
- The 3 "anchored" community posts (`ANCHORED_POSTS` in `content/community.js`, keyed by fixed literal dates like `KERUH_DATE`) retain identical `judul`/`isi`/`createdAt` — only their rng-derived `id` (uuid) changed, same as every other regenerated row across all 14 downstream files (only UUIDs and non-anchored random content shifted, never clinical values or anchor relationships).

Did not attempt to refactor the generator to give every section its own independently-seeded RNG (which would be the "true fix" for this class of coupling) — that is a larger structural change across the entire ~1200-line generator, out of this plan's scope, and carries its own regression risk to a file that was just verified end-to-end in the prior quick task (260706-q0g). Flagging as a known architectural property rather than fixing it.
