# Quick Task 260707-98x: Fix push notification appearing on the editing device itself

**Originating device now excludes itself from the "Pengingat Diperbarui" cross-device push via a self-reported `X-Push-Endpoint` header, filtered server-side against the user's active subscriptions.**

## Performance

- **Duration:** ~15 min
- **Tasks:** 2/2 completed
- **Files modified:** 8

## Accomplishments

- Backend `sendToAllDevices` now accepts an optional `excludeEndpoint` and filters it out via a new exported pure helper `excludeOriginator`, with the "skip fan-out" guard correctly reordered to check the *post-exclusion* target list (previously it checked pre-exclusion `subs.length`, which would have still tried to fan out to a sole self-device).
- `updateReminder` reads the new `X-Push-Endpoint` request header and passes it through to `sendToAllDevices`.
- Frontend `getCurrentPushEndpoint()` added to `pushClient.ts`, reading the current device's own push subscription endpoint (or `null` if unsupported/unsubscribed), and wired into all 4 reminder-update call sites (toggle + CAPD/HD/Medication edit forms).
- With push not subscribed, the header is simply omitted and behavior is byte-for-byte identical to before the fix.

## Decisions Made

- Followed the plan's interfaces exactly: `excludeOriginator` is a pure function, `sendToAllDevices`'s third param is optional, and the guard checks `targets.length === 0` (post-exclusion), not the pre-exclusion `subs.length`.
- `getCurrentPushEndpoint()` calls are only made in edit mode (`isEditMode` guard) on the 3 form components — no wasted subscription lookups on create, matching "Only attach the header on update (PATCH) paths, not on create (POST) paths."
