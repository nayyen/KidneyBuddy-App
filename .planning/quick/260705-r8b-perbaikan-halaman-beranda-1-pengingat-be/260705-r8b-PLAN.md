  <name>Task 1: Fix Pengingat Berikutnya stale-revert on uncheck (bug 1)</name>
  <files>backend/src/repositories/reminderSchedule.repository.ts</files>
  <action>
    In `findNextUpcoming`, a reminder whose today-slot time has already passed must
    NEVER surface in Pengingat Berikutnya again — not even as tomorrow's occurrence —
    regardless of confirmation status. Today it "belongs to" the Obat/Cuci Darah Hari
    Ini list. The current bug: when a past-due reminder is unchecked it leaves
    `confirmedIds`, so `active` re-includes it; `upcomingToday` (jamPengingat >=
    currentTime) is empty for it, so `findNext` falls through to the `tomorrowReminders`
    branch and returns it as the next slot for tomorrow — making it "revert" into
    Pengingat Berikutnya.

    Fix inside the `findNext` closure: before computing the tomorrow fallback, exclude
    from the tomorrow candidate set any reminder that is ALSO active today
    (`hariAktif` includes `todayDay`) AND whose `jamPengingat < currentTime` (its today
    slot has already passed). Such reminders remain owned by today's list for the rest
    of the day. Only reminders whose today slot has NOT passed, or that are genuinely
    tomorrow-only, may populate the tomorrow branch. Keep the existing same-slot grouping
    (return all reminders sharing the earliest `jamPengingat`). Do not change the
    upcoming-today branch or the `confirmedIds` exclusion — those are correct.

    Add a short comment referencing quick-260705-r8b explaining that a past-due-today
    reminder is intentionally excluded from the tomorrow fallback so an uncheck cannot
    re-qualify it as "next".
  </action>
  <verify>
    <automated>cd backend && npx tsc --noEmit</automated>
  </verify>
  <done>An unchecked past-due reminder no longer appears under Pengingat Berikutnya (it stays in Obat/Cuci Darah Hari Ini); a genuine tomorrow-only reminder still appears; tsc clean.</done>
