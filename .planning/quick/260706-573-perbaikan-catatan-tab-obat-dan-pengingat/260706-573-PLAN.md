<task type="auto">
  <name>Task 2: Make uploaded medication photos actually render on both surfaces</name>
  <files>frontend/components/catatan/MedicationLogItem.tsx, frontend/components/pengingat/ReminderDetailOverlay.tsx, frontend/next.config.ts, backend/src/app.ts, backend/src/lib/upload.ts, backend/src/services/medicationLog.service.ts</files>
  <action>
Both display components are already coded to render `<Image src={`${API_BASE}${fotoObat}`} ... onError={() => setPhotoFailed(true)} />` and next.config.ts already allow-lists `/uploads/**` on the backend origin. Because `onError` SILENTLY HIDES a failed image, a broken link looks identical to "no photo". So this is a diagnose-then-fix task — do NOT assume; reproduce and trace the actual broken link in BOTH paths independently before editing.

Reproduction + diagnostic ladder (run the dev stack, upload a JPEG/PNG to an obat reminder via the /pengingat medication form first):

1. DATA — confirm the API returns a non-null path:
   - /pengingat path: `GET /api/reminders` must include `fotoObat: "/uploads/medication-photos/<file>"` on the reminder. `listReminders` returns raw rows, so the column should pass through — verify it does.
   - /catatan path: `GET /api/medication-log/today` — `medicationLog.service.ts` attaches `fotoObat` from the owning reminder via `findActiveObatByUser` (aktif=true only). Confirm the confirmed/pending log for a reminder-with-photo carries `fotoObat`. If the owning reminder is disabled (aktif=false) its logs currently lose the photo — if that edge case is the failure, broaden the lookup so a log's photo resolves regardless of the reminder's aktif toggle.

2. URL — open `${API_BASE}/uploads/medication-photos/<file>` directly in the browser:
   - If it 404s → static-serving path mismatch. `app.ts` serves `express.static(path.join(__dirname, "../uploads"))` while `upload.ts` writes to `UPLOAD_DIR` (default `/app/uploads/medication-photos`). Inside Docker `__dirname`→`/app/dist` so `../uploads`→`/app/uploads` and paths align; outside Docker they diverge. Fix by making the static-serve directory and the multer write directory resolve to the SAME base (e.g. serve from the resolved `UPLOAD_DIR`'s parent, or point both at one shared env-configurable path). Ensure the served route stays `/uploads/medication-photos/<file>` so existing DB paths and remotePatterns keep matching.
   - If it 200s but the `<Image>` still hides → next/image is rejecting the origin. Confirm `API_BASE` (`NEXT_PUBLIC_API_URL`) origin exactly matches an entry in `next.config.ts` remotePatterns (protocol/host/port/pathname). Fix the remotePattern to cover the real deployed origin. If next/image optimization remains the blocker for this static file server, fall back to a plain `<img>` with the fully-qualified src on BOTH components (keep the onError-hide-on-broken behavior).

3. Apply the minimal fix that makes the photo visibly render in BOTH the /catatan obat-tab detail overlay and the /pengingat reminder detail overlay. Keep the graceful onError fallback in both. Only edit the files actually implicated by the diagnosis (do not change files that the ladder proves are already correct).
